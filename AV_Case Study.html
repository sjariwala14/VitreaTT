SingHealth Post Processing Advanced Viewer System

Project Background

  • Existing Vitrea solution reached End-of-Life/End-of-Support (EOL/EOS)
  • Initial deployment in 2011 with V6.7.4 (separate instances)
  • Current solution across SGH and CGH V6.7.4 (separate instances)

Scope of Work

  • Implement enterprise-wide multi-modality 3D/4D advanced visualization solution
  • Key focus on cybersecurity
  • Upgrade from v6.7.4 to latest version v7.15.3
  • Consolidation of all applications across 35 SingHealth concurrent licenses
  • Deploy new clinical solutions
  • Integrate with multiple "best in breed" thin slice archiving solutions and existing PACS
  • Adhere to evolving cybersecurity guidelines (Synapxe secure Vitrea architecture)
  • Design to cater for institutional expansion

Technical Specifications

  • Ports Used:
    • HTTPS (TCP 443) for web client access
    • RDP (TCP 3389) for remote desktop sessions
    • DICOM (TCP 104, 11112) for PACS integration
    • SQL Server (TCP 1433) for database communication
  • Network Requirements:
    • Minimum 1Gbps network backbone
    • Separate VLAN for Vitrea components (VLAN ID: 210)
    • Jumbo frames (MTU 9000) enabled for NAS communication

Implementation Timeline

Phase 1 (Q1 2023): NHC, NCC and SGH deployment
Phase 2 (Q2 2023): CGH deployment
Phase 3 (Q3 2023): KKH deployment and system-wide integration
Ongoing: Monthly security audits and quarterly performance tuning

Proposed Solution Summary

  • Consolidated delivery across three separate sites
  • Separately installed for three institutions at their local data center
  • Integrated with 'Thin Slice Life Cycle Management system'
  • Deployed on hospital network on separate VLAN
  • Management Server broken into multiple parts for security
  • Phased deployment starting from NHC, NCC and SGH, followed by CGH and KKH
  • All HTTPS/TLS based Vitrea client delivery using HTML5 of RDP sessions

Architecture Breakdown - Phased Across SingHealth

Components deployed within three different datacenters:

  • SGH Campus (including SGH, NCCS (Cancer center) and NHCS (Heart Center))
  • CGH
  • KKH

Primarily integrated with 'Thin Slice Life Cycle Management system' to retrieve images for post processing.

End-to-End Viewing Workflow

  1. Client (via HTTPS web port) communicates with the RDP broker
  2. RDP broker communicates with the Management server
  3. Management server communicates with Vitrea Application server (1:n) and also with MS SQL Server on separate instance
  4. SQL Server identifies the studies on the NAS
  5. NAS returns the studies information to the Application server
  6. Application server provides session information to Management server and RDP broker
  7. RDP broker then returns the study to client for display

Component Breakdown

Management Server: Heart of the system, all DICOM interactivity with outside entities goes through the Management node.

MS SQL Database Server: Microsoft SQL Standard used to index information about hosted studies and track interactions.

RDP Broker Server: Installed on a separate server on a separate VLAN, communicates directly with Application server.

All DICOM images from Thin Slice Server C-store DICOM objects to Management node of Vitrea Advanced.

DICOM updates occurring in Vitrea Advanced are C-stored back to Thin slice server for permanent archiving.

Network Configuration Details

VLAN Configuration:

  • VLAN 210: Primary Vitrea network (10.210.0.0/24)
  • VLAN 211: RDP Broker isolation (10.211.0.0/24)
  • VLAN 212: Database network (10.212.0.0/24)

Firewall Rules:

  • Allow TCP 443 from clinical workstations to RDP Broker
  • Allow TCP 3389 from RDP Broker to Application Servers
  • Allow TCP 1433 from Management Server to SQL Server
  • Allow TCP 104,11112 from PACS to Management Server
  • Deny all other traffic between VLANs

QoS Configuration:

  • Priority 1: RDP traffic (DSCP 46)
  • Priority 2: DICOM transfers (DSCP 34)
  • Priority 3: Database replication (DSCP 18)

Option A: 30 Local ESS Instances (Recommended)

  • 30 ESS instances (one per hospital)
  • WAN connection: 30 MBPS
  • Sync between local and global PACS
  • Radiologist workflow:
    1. Select study in local PACS worklist
    2. Global Vitrea instance executed via hyperlink (SSO)
    3. Thin client required on local system for RDP

Pros: Better performance, local processing

Cons: Higher deployment complexity

Option B: One Enterprise Instance (Not Recommended)

  • Single global Vitrea Management Server (Virtual)
  • 5 x 6-User Application Servers (30 CCU total)
  • WAN connection to all hospitals
  • Risk: Bandwidth constraints and performance issues

Note: CMSM team aware of concerns but prefers this option

Option C: Grouped Enterprise Instance

  • Geographically group hospitals (North, South, Center, East, West)
  • Each group gets:
    • 1 ESS + PACS integration + Domain controller (6 CCU)
    • Or 1 Management server + 2x6 User Application servers (12 CCU)
  • Real-time sync with global PACS

Advantage: Balanced approach between Options A and B
