What is DICOM?

Digital Imaging and Communications in Medicine

Detailed specification for transferring medical images and related information between computers

Key Points:

  • It is a standard, not a product
  • Enables cross-vendor connectivity
  • Facilitates interchange of medical information
  • Sets foundation for interoperability between devices
  • Complies to ISO 12052:2017

What DICOM Defines:

  • Data format (images, orders, exams, reports)
  • Tags
  • Communication protocol
  • Portable media
  • Security

DICOM Standard Structure:

  • Parts 1 through 22 (as of October 2023)
  • Parts 9 and 13 were abolished
  • Supplements add information to the standard
  • Correction Proposals (CPs) correct the standard

History of DICOM

1983

American College of Radiology (ACR) & National Electrical Manufacturers Association (NEMA) form a joint committee to develop a standard interface between diagnostic imaging equipment

1985

ACR-NEMA Version 1.0 released

1988

ACR-NEMA Version 2.0 released

1993

DICOM Version 3.0 started - Applicable to a network environment

Revision Process:

DICOM Committee meets 5 times annually to vote on CPs and Supplements. Changes are incorporated into DICOM standards upon vote approval by DICOM Committee members.

DICOM Communication Protocol

Main Concepts:

SCU (Service Class User)

Request side in DICOM Communication

SCP (Service Class Provider)

Service providing side in DICOM Communication

Protocol Steps:

  1. Association Request (A-ASSOCIATION-RQ)
  2. DICOM Message/Data Transfer (C-STORE, C-FIND, etc.)
  3. Association Release (A-RELEASE-RQ/RSP)

Service Classes:

  • Verification
  • Storage
  • Storage Commitment
  • Query/Retrieve
  • Modality Worklist Management (MWM)
  • Modality Performed Procedure Step (MPPS)
  • Print Management

Required Information:

  • Data Encoding (Big/Little Endian, Compression)
  • AE Title (Application Entity Title) - max 16 bytes, unique on network

DICOM Data Format

DICOM files contain both metadata and image pixel data in a set.

Metadata includes:

  • Patient information
  • Examination information
  • Imaging information

Data Elements:

Each piece of information is identified by a tag consisting of Group and Element numbers (hexadecimal).

// Example DICOM Tags
(0010,0010) Patient's Name
(0010,0020) Patient ID
(0010,0030) Patient's Birth Date

Value Representation (VR):

Defines the format of the Value Field. May be explicit or implicit.

VR Types:

  • Fixed formats: AE, PN, UI
  • Math formats: FD, UL
  • Character strings: CS, ST
  • Other: OB, OW

Explicit vs. Implicit VR:

Explicit VR: VR is sent in the data element

Implicit VR: VR is not sent (defined in PS 3.6)

Transfer Syntax:

Encoding used to send data over the network:

  • Implicit VR Little Endian (Default for Network)
  • Explicit VR Little Endian (Default for Media Storage)
  • Explicit VR Big Endian
  • Others with compression (Lossless, Lossy)

DICOM Data Structure

Hierarchical Structure:

Patient
└ Study
　　└ Series
　　　　└ Image

Key Concepts:

IE (Information Entity)

Defines properties and attributes related to classifications (Patient, Study, Series, Image)

IOD (Information Object Definition)

Combination of IEs for each modality (e.g., CT Image IOD, MR Image IOD)

Module Attributes:

Each IE consists of modules with specific attributes:

Attribute Types:

  • Type 1: Mandatory - must be included with a value
  • Type 1C: Conditional - required if conditions are met
  • Type 2: Mandatory but can be empty
  • Type 2C: Conditional Type 2
  • Type 3: Optional

Character Sets:

DICOM supports various character sets for text representation:

  • Single-Byte Character Sets without Code Extensions
  • Single-Byte Character Sets with Code Extensions
  • Multi-Byte Character Sets with Code Extensions
  • Multi-Byte Character Sets without Code Extensions

DICOM Services

Service Classes:

Set of Service Class definitions which provide an abstract definition of actions applied to Information Objects.

Most Common Services:

  • Verification
  • Storage
  • Query/Retrieve
  • Print Management
  • Modality Worklist Management (MWM)
  • Modality Performed Procedure Step (MPPS)
  • Storage Commitment
  • Instance Availability Notification

Service Object Pair (SOP):

Combination of IOD (Information Object) and Service Class.

SCU (Service Class User)

Client that requests a service

Example: CT Scanner sending images

SCP (Service Class Provider)

Server that provides a service

Example: PACS receiving images

DICOM Communication Protocol:

  1. Association Request (A-ASSOCIATION-RQ)
  2. DICOM Message/Data Transfer (C-STORE, C-FIND, etc.)
  3. Association Release (A-RELEASE-RQ/RSP)

DICOM Structured Reporting

Structured Reporting (SR):

DICOM SR provides a standardized way to create, exchange, and manage structured medical reports.

SR Key Concepts:

  • Document Root Node: Top-level container for the report
  • Content Nodes: Individual pieces of information (text, codes, measurements, etc.)
  • Relationship Types: Defines how content nodes relate to each other
  • By-value vs. By-reference: How content is included in the report

SR Information Model:

Document Root Node CONTAINER "Chest X-ray" (Document Title)
├── Has Observation Context
├── Content Node PNAME "Recording Observer = Smith^John^^Dr^"
├── Content Node CODE "finding = mass"
├── Heading Node CONTAINER "Conclusions"
│ └── Content Node CODE "conclusion = probable malignancy"
├── Heading Node CONTAINER "Specific Image Findings"
│ ├── Content node SCOORD "best illustration of findings"
│ ├── Leaf Node IMAGE "Baseline"
│ └── Leaf Node NUM "diameter = 1.3 cm"
└── Content Node UIDREF "Study Instance UID of Evidence = 1.2.3.4.5.6.7.100"

Codes in SR:

SR uses standardized codes to ensure consistency and interoperability:

Code Systems:

  • SNOMED: Systematized Nomenclature for Human and Veterinary Medicine
  • LOINC: Logical Observation Identifiers, Names and Codes
  • CPT: Current Procedural Terminology
  • ICD-9/10: International Classification of Diseases
  • BI-RADS: Breast Imaging Reporting and Data System

Code Structure:

  • Coding Scheme: Identifier for the code system
  • Code Value: Specific code within the system
  • Code Meaning: Human-readable description
  • Scheme Version: Version of the coding system

SR Benefits:

  • Standardized reporting format across systems and institutions
  • Structured data that can be processed by computers
  • Improved data mining and research capabilities
  • Enhanced integration with EHR systems

DICOM Workflow Management

Key DICOM Workflow Services:

Modality Worklist

Provide demographics and order details

→

MPPS

Logging/tracking of procedure status

→

Storage Commitment

Confirmation of data storage

→

Instance Availability

Notification of data availability

→

Workflow Complete

Integrated acquisition workflow

Modality Worklist (MWL):

Functionality:

  • Provides demographics and order details to modalities
  • SCP: Usually RIS, sometimes PACS or third-party box
  • SCU: Usually modality or imaging system
  • Query can include filters (date/time, patient name, accession #, etc.)

Modality Performed Procedure Step (MPPS):

Status Tracking:

  • In Progress: Indicates procedure step is in progress
  • Completed: Indicates procedure step has been completed
  • Discontinued: Indicates procedure step was discontinued
  • Includes tracking attributes (Accession#, SPS ID, Study UID, etc.)

Storage Commitment:

Functionality:

  • Provides confirmation of data storage
  • SCP: Usually PACS
  • SCU: Usually modality, workstation, or another PACS
  • Request specifies list of Data Object UIDs
  • Response confirms storage or provides failure reason

Instance Availability Notification:

Functionality:

  • Provides notification of availability of imaging data
  • SCU: Usually PACS or other devices supporting Retrieve
  • SCP: Usually RIS, reporting system, or workflow manager
  • Notification includes object UIDs and availability status

Benefits of DICOM Workflow Management:

  • Reduced data entry errors and time
  • Accurate, detailed data on performed steps
  • Reduces lost data through storage commitment
  • Enables timely reading workflow and fast reporting
  • Improves overall efficiency in medical imaging departments

Summary

Key Points:

  • DICOM is a standard that enables cross-vendor connectivity for medical imaging
  • Defines data formats, communication protocols, and services
  • Uses a hierarchical structure with IODs, IEs, Modules, and Attributes
  • Relies on UIDs for global uniqueness of objects
  • Uses Service-Object Pairs (SOPs) to define interactions between systems
  • Structured Reporting provides standardized reporting format
  • Workflow Management services optimize imaging department operations

Benefits of DICOM:

  • Interoperability between vendors
  • Consistent data representation
  • Standardized communication
  • Extensibility for new modalities and use cases
  • Improved workflow efficiency
  • Enhanced data mining and research capabilities
